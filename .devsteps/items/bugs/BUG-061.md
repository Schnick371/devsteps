## Security Classification
Priority: P1 (Critical)

## Problem
`chat.plugins.enabled: false` is not explicitly blocked in the DevSteps workspace settings. External plugin hooks execute shell commands on the host system. If a malicious or misconfigured plugin is loaded while this setting is absent, it can:
- Execute arbitrary shell commands
- Read and exfiltrate `.devsteps/` data (work items, metadata, credentials in tags/descriptions)
- Modify workspace files without user awareness

## Root Cause
`.vscode/settings.json` does not enforce `chat.plugins`: constraints. The workspace trusts VS Code default behaviour rather than explicitly disabling unsafe plugin hooks.

## Fix
Add to `.vscode/settings.json`:
```json
"chat.plugins.enabled": true,
"chat.plugins.allowHooks": false
```
Or at minimum document the risk and add a comment in `settings.json` noting that `chat.plugins.paths` entries must be reviewed before enabling hooks.

Also audit existing `plugin.json` (STORY-199 output) to ensure no `hooks` capability is declared unless explicitly reviewed.

## Acceptance Criteria
- `.vscode/settings.json` explicitly addresses plugin hook security
- DevSteps `plugin.json` manifest (when created) contains no `hooks` declarations unless security-reviewed
- SECURITY.md updated with a note about plugin hook risks
- Gate-reviewer confirms no shell-executable hook paths are present

## Files Affected
- `.vscode/settings.json`
- `SECURITY.md`
- `plugin.json` (when created by STORY-199)
