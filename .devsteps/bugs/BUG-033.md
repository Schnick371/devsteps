# Bug: CLI --version displays incorrect hardcoded version

## Problem

CLI `--version` command shows hardcoded "0.1.0" regardless of actual package version:

```bash
$ devsteps --version
0.1.0  # Should show 0.7.0-next.2
```

**Affected Locations:**
1. `packages/cli/src/index.ts` line 19: `program.version('0.1.0')`
2. `packages/cli/src/commands/init.ts` line 48: `version: '0.1.0'`

## Impact

- **User confusion**: Published version is 0.7.0-next.2, but CLI reports 0.1.0
- **Debugging difficulty**: Cannot verify installed version matches expectations
- **Professional appearance**: Established CLI tools read version dynamically from package.json
- **Maintenance burden**: Version must be manually updated in multiple files

## Expected Behavior

`devsteps --version` should return actual version from `package.json`:
- Development: Current workspace version
- Published: npm package version

## Root Cause

Hardcoded string literals instead of reading from package.json metadata.

## Reproduction

```bash
# Install latest pre-release
npm install -g @schnick371/devsteps-cli@next

# Check version
devsteps --version
# Actual: 0.1.0
# Expected: 0.7.0-next.2
```

## Technical Context

**Best Practice (from research):**
- Commander.js supports `.version()` reading from package.json
- TypeScript ESM modules can import package.json with `import.meta` or `assert { type: 'json' }`
- Industry standard: Read version dynamically at runtime

**Example Implementations:**
- npm CLI
- Next.js CLI  
- Vercel CLI
- All use dynamic version reading
