# Bug: Init überschreibt .gitignore vollständig

## Problem

Beim Ausführen von `devsteps init` wird die `.gitignore` Datei **komplett überschrieben**, was zu Datenverlust führt.

### Aktuelles Verhalten

**Beide Init-Implementierungen** (CLI + MCP Server) überschreiben `.gitignore`:

```typescript
// packages/cli/src/commands/init.ts:79-86
// packages/mcp-server/src/handlers/init.ts:85-93
const gitignore = `.devsteps/
node_modules/
dist/
*.log
.env
.DS_Store
`;
writeFileSync(join(projectPath, '.gitignore'), gitignore);
```

### Betroffene Szenarien

1. **Bestehende Projekte**: Projektspezifische ignore-Regeln gehen verloren
   - Build-Artefakte
   - IDE-Konfigurationen
   - Plattformspezifische Dateien
   - Sensitive Daten

2. **Mehrfaches Init**: Nicht-idempotent (wiederholtes Init ist sicher)

3. **Team-Workflows**: Zerstört Team-Konventionen

### Reproduktion

```bash
# Bestehendes Projekt mit .gitignore
echo "coverage/" > .gitignore
echo "*.tmp" >> .gitignore

# DevSteps Init
devsteps init my-project

# Result: .gitignore enthält nur DevSteps Standard-Regeln
cat .gitignore
# .devsteps/
# node_modules/
# ...
# ❌ coverage/ und *.tmp sind weg!
```

## Expected Behavior

`.gitignore` sollte **append-only** sein:
- Nur `.devsteps/` Eintrag hinzufügen (falls noch nicht vorhanden)
- Existierende Regeln erhalten
- Idempotent (mehrfaches Init sicher)

## Impact

- **Severity**: High (Datenverlust)
- **Frequency**: Jedes Init in bestehendem Projekt
- **User Experience**: Vertrauensverlust, manuelle Reparatur nötig

## Affected Files

- `packages/cli/src/commands/init.ts` (Zeile 78-86)
- `packages/mcp-server/src/handlers/init.ts` (Zeile 85-93)
